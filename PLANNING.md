# 灘校生徒会サイト 計画書（2026-06-28時点）

このドキュメントは、現状のシステム評価・セキュリティ確認結果・今後の機能提案をまとめたもの。
コードの変更履歴はgit log、DBスキーマの変更履歴は`supabase/migration_v*.sql`を参照。

## 1. システム評価

### 1.1 構成
- フロントエンド: 静的HTML 3枚（`index.html`=生徒用, `admin.html`=管理者用, `post.html`=投稿詳細）、フレームワーク無し・ビルドステップ無しのVanilla JS
- バックエンド: Supabase（Postgres + Auth(Google OAuth) + Storage）、Row Level Security(RLS)で認可を制御
- 外部連携: Cloudflare Pages Functions（`functions/api/*.js`）経由でLINE Messaging APIと連携
- ホスティング: Cloudflare Pages（GitHub `atsu427/student-council` の `main` ブランチを自動デプロイ）

### 1.2 良い点
- RLSが全テーブルで有効化され、`is_admin()`関数による権限判定が一貫して使われている
- LINE関連の秘密情報（Channel Secret / Access Token / Service Role Key）はCloudflare Functionsの環境変数のみに置かれ、クライアントに露出していない
- LINE Webhookの署名検証（HMAC-SHA256）が実装済み
- 投稿本文・生徒名・メールアドレス等のユーザー入力は、ほぼ全箇所で`escapeHtml()`を通してから表示しており、XSS対策の意識は概ね行き届いている（後述の通り、今回の確認で漏れを数件発見・修正）

### 1.3 技術的負債・課題
| # | 内容 | 影響 |
|---|---|---|
| 1 | `escapeHtml`・Supabase接続設定・`showError`等の共通コードが index.html / admin.html / post.html に**3重に重複**している | 1箇所の修正を3ファイルに反映し忘れるリスク（今回のXSS修正でも3ファイル分の対応が必要だった） |
| 2 | 自動テストが一切無い | 機能追加のたびに既存機能の動作確認が手動・目視のみになっている |
| 3 | dev/staging環境が無く、`main`への push が即本番反映 | 検証なしに本番へ反映される。Cloudflare Pagesはブランチ単位のプレビューURLを自動生成できるが未活用 |
| 4 | 管理者の初期付与が`ADMIN_EMAIL`のハードコードに依存 | そのメールアドレスが使えなくなった場合、DBを直接編集する以外に管理者を復旧する手段が無い |
| 5 | 投稿一覧・生徒一覧などが「全件取得してクライアント側でフィルタ」する設計 | 投稿数・生徒数が今後大きく増えると初回読み込みが遅くなる可能性 |

## 2. セキュリティ確認

### 2.1 今回の確認で発見し、修正済みの項目
1. **カテゴリ名のXSS（未エスケープ）** — `index.html`の投稿一覧、`post.html`の詳細ページ、`admin.html`の投稿一覧で、カテゴリ名をHTMLに埋め込む際エスケープしていなかった。カテゴリは管理者のみ作成可能なため悪用には管理者権限の侵害が前提だが、防御の多重化として修正
2. **`escapeHtml`が引用符(`"` `'`)をエスケープしていなかった** — `class="category-tag ${...}"`のような属性値コンテキストでは、`<` `>` だけでなく引用符のエスケープも無いと属性を抜け出してイベントハンドラを注入できてしまう。3ファイルとも修正
3. **反射型XSS（今回のログイン修正で新たに混入させてしまった不具合）** — Google認証コールバックのエラー内容(`error_description`)をURLから読み取り`innerHTML`に直接渡していたため、悪意のあるURL（`?error_description=<img onerror=...>`）を踏ませることでスクリプト実行が可能だった。`escapeHtml`を通すよう修正済み

### 2.2 現状の対策（既にできている部分）
- 全テーブルRLS有効化、書き込み系は基本`is_admin()`必須
- Storageの書き込み（upload/update/delete）も`is_admin()`必須、読み取りはログインユーザーのみ
- CSRF: Cookieベースの認証ではなくJWTをAuthorizationヘッダーで送る方式のため、古典的なCSRFの影響を受けない構成

### 2.3 残っているリスク・推奨対応
優先度順。

**高（本番公開前に対応推奨）**
- `ALLOWED_DOMAIN`が3ファイルとも`null`（ドメイン制限なし）になっている。現状は**Googleアカウントを持つ誰でも**ログインしてmembersに登録され、ロール未指定のお知らせ・意見箱・DMにアクセスできる状態。本番運用前に`'stg.nada.ac.jp'`へ戻すこと（各ファイルに「本番に戻す場合は」というコメントを残してあるが、戻し忘れている）

**中**
- Storageの添付ファイルは`post-files`バケットの「Public」がONであることを前提にしており、公開URLを知っていれば**ログイン不要・ロール制限も無視して**ファイルを閲覧できる。ロール限定のお知らせに添付した資料が、URLが漏れた場合に対象外の生徒や外部の人にも見られる可能性がある。重要な添付ファイルがある場合は、署名付きURL（一定時間だけ有効なURL）方式への変更を検討
- LINE連携コード（8桁）に有効期限・使い切り制限が無い。コードを他人に見られた場合、その人が自分のLINEを他人のアカウントに紐付け直せてしまう（通知の横取り）。リンク成功後にコードを再発行する、またはWebサイト側でも紐付け完了を明示するなどの対策を検討

**低（手間をかけずに直せる改善）**
- LINE Webhookの署名比較が`===`（非定数時間比較）。タイミング攻撃の理論的リスクはあるが実害は低い
- `line-notify-post.js`のPostgRESTフィルタ文字列がメールアドレス・ロール名を直接埋め込んでいる（入力元は管理者のみなので実害は低いが、念のためエスケープ/パラメータ化を検討）
- 投稿削除・全データ削除など破壊的操作の実行者・実行日時を記録する仕組みが無い（操作ログが無い）

## 3. 今後の機能提案・改善ロードマップ

### 3.1 すぐ着手できる小粒な改善
- 投稿一覧・生徒一覧のページネーション（投稿数・生徒数が増えた時の表示速度対策）
- 添付画像のアップロード時のリサイズ/圧縮（Storage容量・表示速度の節約）
- 管理者の操作ログ（誰が・いつ・何を削除/編集したか）を`admin_logs`テーブルに記録し、危険操作（全データ削除等）の追跡可能性を確保
- スマホのホーム画面に追加できるPWA化（`manifest.json`+アイコン）。LINEからの遷移や日常的な利用が多い前提なら効果が大きい

### 3.2 中期（権限・運用の幅を広げる）
- 管理者ロールの細分化（例: 投稿のみ操作できる「広報担当」、生徒管理のみの「庶務担当」など）。現状`is_admin()`は全権限が一つに紐づいているため、生徒会内の役割分担に対応できない
- 投稿の編集履歴（誰が・いつ・どう変更したか）を保存し、「訂正」通知だけでなく履歴自体も生徒に見せられるようにする
- 行事予定をカレンダー形式で見られるビュー（時間割とは別に、文化祭・説明会などのイベント管理）
- LINE以外の通知手段（Web Push通知）の追加。LINE未連携の生徒への到達率向上

### 3.3 長期（アーキテクチャ刷新を検討する場合）
- 3ファイルに分散している共通JS（Supabase初期化・escapeHtml等）を `common.js` に切り出す、または軽量なビルドステップ（Viteなど）を導入してコード重複を解消
- 最低限のE2Eテスト（ログイン→投稿作成→削除、のような主要フロー）をPlaywright等で自動化し、CIで`main`へのpush前に実行
- Cloudflare Pagesのブランチプレビュー機能を活用したstaging運用（featureブランチ→プレビューURLで確認→main統合）

## 4. 次にやること（推奨優先順位）
1. `ALLOWED_DOMAIN`を本番ドメインに戻す（本番公開前に必須）
2. 添付ファイルのアクセス制御方式を検討する（公開バケットのままで許容するか、署名付きURLに変える）
3. 操作ログ（誰が何を削除/編集したか）の実装を検討する
4. 共通JSの切り出し（次に大きな機能追加をする前に着手すると以後の修正が楽になる）
